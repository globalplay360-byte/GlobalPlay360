1 Per què dues capes

Stripe pot fallar al motor (webhook, claims, filtre past_due) o a la UI (Checkout, Portal, banners). Provar només 4242 deixa els errors a cegues.

Capa 1 · Motor

Codi, CF, rules, sync Firestore. Ràpid, repetible, sense obrir Checkout.

Capa 2 · Interfície

Navegador real + targetes TEST a Stripe Checkout / Portal. Avui: manual (Playwright = següent iteració).

2 Què pot fer l’agent vs tu

L’agent pot: branca + npm run dev, verificar el fix past_due al codi, tests de functions, deixar protocol/guia, actualitzar HANDOFF.
Tu (navegador) has de: les 6 proves de targeta a Stripe Checkout hosted. L’agent no pot completar sol el formulari de pagament de Stripe (3DS, iframe, sessió real) sense un E2E Playwright dedicat encara no escrit.

3 Preparació

  1. Branca: fix/past-due-billing-portal
  2. Dev: npm run dev → http://localhost:5173
  3. Stripe Dashboard → Test mode ON
  4. Compte sense sub trialing/active/past_due (si cal: cancel immediately a Stripe)
  5. Checkout: caducitat 12/34 · CVC 123 · CP 08001

4 Capa 1 — Motor (agent)

Checklist estàtica (pre-N3)

  • Listener inclou past_due + unpaid → stripe.service.ts
  • Billing no redirigeix si cal Portal → BillingPage.tsx
  • Banner + CTA a Pricing → PricingPage.tsx
  • Guard antidoble a la CF (ja a main)
  • npm test a functions/ (billingPolicy)
Truc. La Capa 1 demostra que si arriba un past_due, la UI el pot gestionar. La Capa 2 demostra que Stripe genera aquest estat amb targetes reals de TEST.
